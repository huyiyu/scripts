## Question 1 | HPA autoscale

{{< tabs "题目" "答案" >}}
{{< tab "题目" >}}
在 `autoscale` namespace 中创建一个名为 `apache-server` 的新HorizontalPodAutoscaler(HPA)
- 此 HPA 必须定位到 `autoscale` namespace 中名为 `apache-server` 的现有 Deployment 。
- 将 HPA 设置为每个 Pod 的 CPU 使用率旨在 `50%` 。将其配置为至少有 `1` 个 Pod，且`不超过` `4`个 Pod 。此外，将缩小稳定窗口设置为 `30` 秒。
{{< /tab >}}
{{< tab "答案" >}}
- ***您必须连接到正确的主机。不这样做可能导致零分!!!***: `ssh cka000000`
- **参考链接**:  https://kubernetes.io/zh-cn/docs/concepts/workloads/autoscaling/horizontal-pod-autoscale/
- **使用命令**: `kubectl autoscale -n autoscale deploy apache-server --cpu=50% --min=1 --max=4`
- **设置缩小稳定窗口配置**: 修改 `hpa.spec.behavior.scaleDown.stabilizationWindowSeconds` 值为30。
- (单词记不住可以考虑 1.切成英文版，2.`kubectl explain hpa.spec.behavior.scaleDown`,3.万不得已打开网站查询：)

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: apache-server
  namespace: autoscale
spec:
  behavior:
    scaleDown:
      stabilizationWindowSeconds: "30"
  maxReplicas: 4
  metrics:
  - resource:
      name: cpu
      target:
        averageUtilization: 50
        type: Utilization
    type: Resource
  minReplicas: 1
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: apache-server
```
- **apply上述文件**: kubectl apply -f 1.yaml
- ***做完退出主机,不这样做可能导致零分!!!***:`exit`

## Question 9 | Gateway

{{< tabs "题目" "答案" >}}
{{< tab "题目" >}}
将现有 Web 应用程序从 Ingress 迁移到 Gateway API。您必须维护 HTTPS 访问权限。注意：集群中安装了一个名为 nginx 的 GatewayClass 。

- 首先，创建一个名为 `web-gateway` 的 Gateway ，主机名为 `gateway.web.k8s.local` ，并保持现有名为 `web` 的 Ingress 资源的现有 TLS 和侦听器配置。
- 接下来，创建一个名为 `web-route` 的 HTTPRoute ，主机名为 `gateway.web.k8s.local` ，并保持现有名为 `web` 的 Ingress 资源的现有路由规则。
- 最后，删除名为 `web` 的现有 Ingress 资源。
{{< /tab >}}
{{< tab "答案" >}}
- ***您必须连接到正确的主机。不这样做可能导致零分!!!***: `ssh cka000000`
- **查询gatewayClass**: `kubectl get gatewayclass` 获得名称为 `nginx`
- **查询ingress**: `kubectl get ingress web -o yaml` 关键信息如下：
  - host: ingress.web.k8s.local
  - serviceName: **web**,servicePort **80**,Path: **/**,PathType: **Prefix**
  - tls-secretName: web-cert
- **创建gateway和httpRoute,找模板**: 
  - https://gateway-api.sigs.k8s.io/guides/tls/
  - https://kubernetes.io/blog/2025/11/06/gateway-api-v1-4/

```yml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: web-gateway
spec:
  gatewayClassName: nginx
  listeners:
  - name: listener
    protocol: HTTPS
    port: 443
    hostname: gateway.web.k8s.local
    tls:
      mode: Terminate
      certificateRefs:
      - kind: Secret
        group: ""
        name: web-cert
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: web-route
spec:
  parentRefs:
  - name: web-gateway
  hostnames:
  - "gateway.web.k8s.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: web
      port: 80
```

- **apply 该文件**:`kubectl apply -f 9.yaml`
- **测试访问**: curl https://gateway.web.k8s.local
- **删除旧Ingress**: `kubectl delete ingress web`

## Question 10 | NetworkPolicy

{{< tab "题目" >}}
从提供的 YAML 样本中查看并应用适当的 NetworkPolicy。
**确保选择的 NetworkPolicy 不过于宽松，同时允许运行在 `frontend` 和 `backend` namespaces 中的 `frontend` 和 `backend` Deployment 之间的通信**。
- 首先，分析 `frontend` 和 `backend` Deployment，以确定需要应用的 NetworkPolicy 的具体要求。
- 接下来，检查位于 `~/netpol` 文件夹中的 NetworkPolicy YAML 示例。注意：***请勿删除或修改提供的示例。仅应用其中一个。否则可能会导致分数降低。***

{{< tab "答案" >}}

## Question 14 | Resources cpu/memory

按如下方式调整所有 Pod 资源请求：
1. 将节点资源平均分配给这 3 个 Pod
2. 为每个 Pod 分配公平的 CPU 和内存份额
3. 添加足够的开销以保持节点稳定

请确保，对容器和初始化容器使用完全相同的请求。您无需更改任何资源限制。
在更新资源请求时，暂时将 WordPress Deployment 缩放为 0 个副本可能会有所帮助。

- 所有 Pod 都在运行并准备就绪

{{< tab "答案" >}}

1. **查看节点资源** (以实际考试环境为准)：
   ```bash
   kubectl describe node k8s-master1 | grep -A 5 "Allocatable"
   # 或
   kubectl get node k8s-master1 -o yaml | grep -A 5 allocatable
   ```

   ```
     cpu:                4

   ```

   ```yaml
   resources:
     requests:
       cpu: "1000m"     # 1核 = 4核 ÷ 4 (3 Pod + 系统开销)
       memory: "800Mi"  # 约 2.8GB ÷ 3 - 余量
   ```
   
   > **注意**：如果编辑后提示无法保存（如资源冲突），先保存到文件再 apply：
   > ```bash
   > kubectl -n relative-fawn get deploy wordpress -o yaml > wp.yaml
   > # 编辑 wp.yaml
   > kubectl apply -f wp.yaml
   > ```

4. **扩容 Deployment**：
   ```bash
   kubectl -n relative-fawn scale deploy wordpress --replicas=3
   ```

5. **验证所有 Pod 正常运行**：
   ```bash
   kubectl -n relative-fawn get pod -l app=wordpress
   # 确保所有 Pod 状态为 Running 且 Ready 为 3/3
   ```
